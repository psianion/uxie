# Uxie v1 — Master Deep-Research Report (Synthesis)

**Date:** 2026-06-03
**Status:** Authoritative pre-build synthesis. Read this before writing any code.
**Inputs:** Six adversarially-verified dimension reports (architecture, documentation, testing, logging, tracking, discord-ux). This consolidates the verifier-`keep` and verifier-`revise` (with corrections folded in) recommendations into one picture, drops `reject` items to the appendix, and reconciles cross-dimension overlaps and locked-spec conflicts.
**Scope reminder:** Uxie is the stateless User-plane Discord bot of SUP. v1 = one module `scrypt`, six commands (`/ping`, `/capture` + `#inbox`, `/search`, `/ask`, `/journal`, `/brief`), Bun + discord.js@^14.26 + zod only. Docs-only repo today.

## 1. Executive Summary

The build is unusually well-specified: the Design Spec and Implementation Plan have already locked the hard architecture (folder-convention modules with no registry, three catch sites, stateless per-call clients, deterministic `client_tag`, `withOwnerGate` helper, `lib/env.ts` as the sole `process.env` reader, classic ephemeral embeds). Across six dimensions and 70+ recommendations, the adversarial verifiers found **no recommendation that breaches statelessness, the plane boundary, the owner gate, the minimum-intent rule, or the forbidden discord.js surface**. The research adds value at the seams the spec named-but-did-not-define and at the operational layer (logging hardening, tracking, CI, docs governance) the plan left thin.

Five things the build must internalize:

1. **Three doc-vs-code contradictions must be fixed in the docs first (SUP §13) before any code is written.** (a) The MCP-client dependency: Guidelines §3 + §12.2 mandate `@modelcontextprotocol/sdk` / `StreamableHTTPClientTransport`, but Plan Task 1 locks deps to `discord.js + zod` only, and Design §357/§209 describe a hand-rolled `fetch` + `AbortSignal.timeout(10_000)` `post()`. Hand-rolled is correct (the SDK assumes a long-lived session, the opposite of the per-call rule) — but **Guidelines §3/§12.2 is the stale doc and must be edited to drop the SDK mandate**. (b) The `client_tag` scheme: UXIE-DISCORD-GUIDELINES.md **line 268** (verified by grep — NOT the design spec, which has zero `randomUUID` references) says "`crypto.randomUUID()` for slash commands"; this contradicts the locked deterministic `uxie-<id>` and breaks server-side retry-dedupe and cross-plane grep. (c) The logger field names: Guidelines §18 says `ts` and `interaction_id`, but the locked Task 4 test and Design §5.2 use `t` and `interactionId`.

2. **The error boundary is the spine.** Router listeners must be `async` with a whole-body `try/catch`, and `replyWithError` must never throw (it defensively `.catch()`es its own `editReply`/`reply` to survive a dead interaction, e.g. 10062). This keeps every rejected promise at catch site #1, not the process-level catch site #3 — the discord.js v15 AsyncEventEmitter readiness rule. The single highest-value test in the suite is the parameterized §7.3 error-mapping contract test.

3. **The owner gate lives in the routers, not in command bodies.** `assertOwner(actor, ownerId)` fires in `interaction-router.ts` and `message-router.ts` *before* `deferReply`/`execute`. Command bodies contain no `assertOwner` call. (The testing dimension got this location wrong; the correction is folded in below and is load-bearing for the anti-pattern lint test and the auth test.)

4. **Logging and tracking are one system, not two.** The deterministic `client_tag` doubles as the correlation id: it goes in the per-interaction `log.child()` scope AND as an `X-Correlation-Id` header on every Scrypt call, so `journalctl -u scrypt -u uxie -u para-raid | jq 'select(.clientTag==…)'` traces one request across all three planes. Latency comes free from `performance.now()` deltas emitted per log line — no in-memory counters, no Prometheus, no OTel (all rejected as stateless breaches / dep-budget breaches / Para-RAID's chartered job).

5. **Polish is seams, not features, in v1.** Classic ephemeral embeds + markdown permalinks back to the Scrypt web UI deliver ~90% of the UX. The advanced moves to engrain now are: a uniform `assertOwner→deferReply({flags: MessageFlags.Ephemeral})→editReply` contract, per-command embed builders with cap enforcement, top-N caps + `AttachmentBuilder` overflow (never pagination), and an interaction-router that branches on interaction TYPE with no-op slots for autocomplete/button/modal so every v1.5 feature is a pure addition, not a refactor.

## 2. Per-dimension "Engrain in v1" recommendations

Effort key: S = small, M = medium, L = large. All items below survived adversarial verification as `keep` or `revise` (corrections folded in). Rejected items are in the Appendix.

### 2.1 Architecture

- **(S) Keep folder-convention + per-module `registerScryptIntegration(bot)`; NO `core/registry.ts`, NO `Integration` interface in v1.** Endorses Design §3.2. `index.ts` calls the register function directly; para-raid adds a sibling call. `core/` is extracted only at the module-#2 duplication threshold — do not pre-create it.
- **(S) Define `BotShell` host interface (`registerCommands`, `onMessage`) as the integration seam.** The spec names `registerScryptIntegration(bot: BotShell)` but never defines `BotShell`. It is the *inverse* of the rejected `Integration` interface: the host exposes capabilities the module calls imperatively. Pure interface, no runtime dep.
- **(M) Inject scrypt clients via a per-interaction `ctx` object, not module-level singletons.** Extend `CommandContext` to `{ clientTag, log, scrypt: { mcp, rest } }`. Clients are constructed **once at boot** and passed by reference (construct-once, inject-per-call) — no per-interaction construction, no cross-interaction cache. Decisive for bun-test mockability. *Cross-check (testing dimension): the Plan already does DI via factory closures `buildCaptureCommand(rest)` with `execute(i, ctx)` where `ctx = { clientTag, log }`. The two are compatible — the seam exists; extend the existing `ctx` shape rather than inventing a new `execute(i, deps)` signature. See Open Decision OD-4.*
- **(M) Router listeners are `async` with whole-body `try/catch`; `replyWithError` never throws.** The v15 AsyncEventEmitter rule. Wrap the entire listener body (after early returns) so rejected promises hit catch site #1, not #3. `replyWithError` defensively `.catch()`es its own reply — the one justified nested catch, and it lives in the boundary, not a command.
- **(L) Hand-roll the MCP streamable-http client as a per-call JSON-RPC POST over `fetch` (no SDK).** One private `post(tool, args)` owns bearer auth, `AbortSignal.timeout(10_000)`, JSON-RPC 2.0 envelope, and the ScryptError mappings; every response is zod-parsed, never cast. **Gated on reconciling the Guidelines §3/§12.2 SDK mandate first — see OD-1.**
- **(S) `client_tag` is deterministic id-derived (`uxie-<id>` / `uxie-msg-<id>`), zero local idempotency.** Required for server-side retry-dedupe. **Fix Guidelines §268 first — see OD-2.**
- **(S) `Bun.Glob` for command discovery; pure `buildCommandCollection` that throws on duplicate names.** Separate I/O `loadAllCommands` (glob + import) from the pure, unit-testable `buildCommandCollection`. *Watch: confirm `import.meta.dir`-relative dynamic `import()` of globbed `.ts` works inside `oven/bun:1-alpine` with no build step; fall back to an explicit command barrel if flaky.*
- **(S) Add `withOwnerGate(builder)` helper** applying `setContexts(InteractionContextType.Guild)`, `setIntegrationTypes(ApplicationIntegrationType.GuildInstall)`, `setDefaultMemberPermissions(0n)`. Already mapped to **Plan Task 4.6** (the test asserts `json.contexts`, `json.integration_types`, `json.default_member_permissions === "0"`). `0n` only hides the command in the UI; `assertOwner` remains the real gate.
- **(S) Graceful shutdown:** `process.once("SIGTERM"/"SIGINT") → await client.destroy() → process.exit(0)`. `client.destroy()` (Symbol.asyncDispose in v14.26) closes the gateway cleanly; no drain timeout (no in-flight queue). *Smoke-verify under `oven/bun`, since Bun signal handling differs from Node.*
- **(S) Pick hand-rolled `log.ts` (not pino); `env.ts` as sole `process.env` reader; define orchestrator-stub seam types now.** `OrchestratorJob`/`OrchestratorResult`/`dispatch()` defined in `orchestrator-stub.ts`, importing only `lib/`, never `integrations/scrypt/`.
- **(v2, M) Defer `core/` extraction** (notifier, shared channel resolver, register loop) until para-raid ships with real duplication.

### 2.4 Logging

- **(S) Ship a thin hand-rolled JSON logger, not pino.** pino's value is worker-thread transports (broken/fragile on Bun: oven-sh/bun#23062, #3757); perf is irrelevant at a-few-interactions/day; pino blows the discord.js+zod-only budget. Synchronous `console.log` to stdout = no buffered lines lost on crash. *Folded-in caveat: verify Bun flushes stdout before `process.exit(1)` in the fatal handler — if not, the most important line (the fatal one) is dropped, undercutting the whole "console.log loses nothing" argument. Await the write or flush before exit.*
- **(S) Use `t` as the timestamp key** (the locked Task 4 test asserts `typeof line.t === "string"`). **Correction (verifier):** fix BOTH drifts in the §18 doc edit — `ts→t` AND `interaction_id→interactionId` (the test and Design §5.2 use camelCase). See OD-3.
- **(S) Harden the serializer:** recursive redaction (nested `{headers:{Authorization}}` would leak under top-level-only), cycle-safe stringify (discord.js objects are circular and would throw *inside the logger*, below the boundary — use a `WeakSet`), and ~2KB/value + ~8KB/line truncation. All verified additive against the Task 4 test.
- **(S) Inject base scope `svc:"uxie"`, `ver`, `env`** at boot via the root `child`. Pairs with a journald `tag: uxie` (CONTAINER_TAG) so uxie is filterable two ways out of the shared three-app stream.
- **(S/M) Add a `debug` level + in-code numeric level gate (default `info`); emit a closed event vocabulary** (`boot.*`, `cmd.start`/`cmd.ok(durationMs)`/`cmd.fail`, `owner.deny`, `inbox.capture`/`ack`/`fail`, `scrypt.req`/`res`, `process.fatal`). **Correction (verifier):** `debug` is NOT a spec deviation — Guidelines §18 already lists `debug|info|warn|error`; only the Plan Task 4 skeleton narrowed it. Adding it aligns the skeleton with §18.
- **(S) Log metadata only — never note bodies, query text, or PII beyond snowflakes.** Log lengths/`clientTag`/`kind`/`scryptStatus`. The vault must never end up in the shared, persistent journald stream. *Folded-in: standardize on the field name `err` for serialized errors (contains no redaction substring) so stacks are never accidentally redacted.*
- **(S) One-error-one-line discipline across the three catch sites.** Boundaries log+map; deeper code throws silently; `process.on` is the backstop for escaped v15 rejections only.
- **(S) Propagate `client_tag` as `X-Correlation-Id` on every Scrypt REST + MCP call** (shared with the Tracking dimension — see Cross-cutting §4). Inert until Scrypt echoes it, but harmless and free.
- **(M) `process.fatal` log + `exit(1)` in `index.ts`** process handlers — completes §14.2's third catch site (currently absent in the Plan, per consistency report rows 4-5). Test an escaped rejection.
- **(v1) stdout-only single-line JSON, no ANSI/pretty-printing** — the only format that is grep/jq-clean under BOTH `docker logs` (v1 dev) and `journalctl -o cat` (prod). Pipe through `jq` only as a CLI consumer (`bun run dev | jq`), never in-process. *Lock this, don't leave it open.* *Folded-in: the logger's redaction substring list stays a hardcoded constant in `log.ts` — never read from `env`, to avoid a circular import (`env.ts` imports `errors.ts`; `log.ts` is foundational).*
- **(v1.5, S) Add `LOG_LEVEL` to `lib/env.ts`** + thread into `createLogger` — **the one genuine env-schema conflict; deferred. See OD-3.** Ship an in-code `info`-default gate in v1; the §17.1 doc bump + env field is v1.5.

### 2.6 Discord UX

- **(S) Uniform command contract:** `assertOwner` (in router) → `deferReply({ flags: MessageFlags.Ephemeral })` → single `editReply`. Legacy `ephemeral:true` is deprecated; `deferReply` flags accepts only `MessageFlags.Ephemeral`. **Reuse the existing named primitives** (`withOwnerGate` in `lib/command-builder.ts`, `assertOwner` in `lib/auth.ts`) — do not invent a parallel `_contract.ts`/`Command` file.
- **(M) Per-command embed builders in `lib/embed.ts`** with cap enforcement (`DESC_CAP`, `FIELD_CAP`), one accent color + ok/warn/err, provenance footers, and markdown `[label](noteUrl)` permalinks. **Correction (verifier):** use the **canonical Design §5.2 names** — `captureEmbed`, `searchResultEmbed`, `semanticResultEmbed`, `briefEmbed` — NOT the invented `searchEmbed`/`askEmbed`/`pingEmbed`. `noteUrl()` must take `SCRYPT_SERVER_URL` via `lib/env.ts` (never inline), and the `/n/<path>` scheme is **unconfirmed** — do not hardcode until Scrypt's route is verified (OD-7). Present `DESC_CAP=4000` as uxie's chosen cap, not the Discord 4096 limit.
- **(S) Top-N caps + `AttachmentBuilder` overflow instead of pagination.** Cap `/ask` to 5, `/search` to 10; overflow note bodies >1500 chars to a `.md` attachment (file boundary: `lib/attachment.ts`, not `lib/embed.ts`); rely on permalinks for the rest. Pagination = stateful collector = forbidden. *Folded-in: ephemeral attachments expire with the message — the permalink is the durable artifact, the `.md` is a convenience.*
- **(S) Router branches on interaction TYPE with no-op slots** for `isAutocomplete`/`isButton`/`isModalSubmit`; `Command` interface gains optional `autocomplete?(i)`. Zero runtime cost, turns every v1.5 feature into a pure addition. The no-op branches throw nothing, so the three-catch-site rule holds.
- **(S) Lock `allowedMentions {parse:[], repliedUser:false}` in the client factory** (already wired in Plan Task 9 — this confirms it). Verify inheritance by `editReply` in a router test (surface §7-Q5). Satisfies the no-rapid-streaming and no-cooldown rules.
- **(v1.5, M) Live `/search` autocomplete** backed by `searchNotes(focused, 5)` — perf-gated (only if Scrypt FTS5 p99 < ~800ms; Discord fires per keystroke under a hard 3s budget where `respond()` is the only valid reply, no `deferReply`).
- **(v1.5, L) Components V2** for `/ask` link buttons + `/brief` rich layout — after an `IsComponentsV2|Ephemeral` spike. GA since March 2025; **mutually exclusive** with content/embeds/poll/stickers (mixing = runtime rejection); must live in a separate `lib/components-v2.ts`; cap at 1 Container + ≤2 ActionRows; link buttons need no customId/collector (stateless).

## 4. Cross-cutting decisions (one picture, not six silos)

- **Correlation id = `client_tag` = trace key.** (logging R9 + tracking R1/R4.) The deterministic `uxie-<id>`/`uxie-msg-<id>` is (a) the server-side idempotency key in the ingest body, (b) the per-interaction `log.child({ clientTag })` scope field, and (c) the `X-Correlation-Id` request header. One value, three jobs. The read-side header is gated on Scrypt accepting it; the log-correlation half ships in v1 regardless. This single decision unifies logging and tracking and is why the §268 `randomUUID` drift is a correctness bug, not a style nit.
- **`/ping` health is REST-only and shared by architecture + tracking.** `rest.health()` returns `{ok, reason}` (never throws — the one sanctioned client-side catch). `/ping` enriches with WS ping (null-safe), roundtrip, `Status`, and `process.uptime()`. No MCP probe (Design §6.7). Keep the reply string-based to satisfy Plan Task 13's test, or change the test (OD-6).
- **Embed conventions are shared by docs + discord-ux.** `lib/embed.ts` builders use the **canonical Design §5.2 names**; `docs/COMMANDS.md` is generated from the builder array; the `/capture` embed screenshot in the PR is the de-facto render test (testing) and the embed-design record (docs). One naming source prevents three silos from drifting.
- **The three catch sites are shared by architecture + logging + testing.** Site #1 (interaction-router) and #2 (message-router) log+map; site #3 (`index.ts` `process.on`) logs `process.fatal` + exits. Deeper code throws silently. The async whole-body try/catch (architecture) + the one-error-one-line discipline (logging) + the §7.3 contract test (testing) are three views of the same invariant. The `process.on` site is currently **missing from the Plan** (consistency rows 4-5) — adding it is a v1 must.
- **Owner gate is in the routers, not commands.** Resolves the testing dimension's wrong assumption. Affects the anti-pattern lint, the auth test, and the `replyWithError` reply-vs-editReply branch (non-owner fires pre-defer → `reply`).
- **Dep budget (`discord.js + zod` only) is the binding constraint behind four "no" decisions:** no pino (logging), no OTel/Sentry/prom-client (tracking), no MCP SDK (architecture — pending OD-1), no typedoc in v1 (docs). Every one of these was decided on the dep budget + Bun fragility + statelessness, not on perf.
- **`inbox-handler.ts` is the one non-router file with a legitimate `try/catch`** (to do the ❌ react). The anti-pattern lint must scope `try{` to `commands/*.ts`, and the "three catch sites" framing should note this react-handler catch is part of message-router boundary #2's surface, not a fourth site.

## 5. Open decisions for the team-lead (conflicts to ratify, with recommendations)

All of these require a **doc edit before code** (SUP §13). Ordered by build-blocking severity.

- **OD-1 — MCP client: SDK vs hand-rolled.** Guidelines §3 (line 44) + §12.2 (lines 274, 277) mandate `@modelcontextprotocol/sdk` / `StreamableHTTPClientTransport`; Plan Task 1 locks deps to `discord.js + zod` only; Design §357/§209 describe a hand-rolled `fetch` + `AbortSignal.timeout(10_000)` `post()`. **Recommendation: hand-roll (the SDK assumes a long-lived session, the opposite of the per-call rule), and edit Guidelines §3 + §12.2 to drop the SDK mandate and describe the per-call `fetch` JSON-RPC client.** This is the single biggest unreconciled contradiction (the architecture verifier's only `revise`). *Also unconfirmed and gating the parsers: does the Scrypt streamable-http server accept a cold `tools/call` POST with no `initialize` handshake / `Mcp-Session-Id`? If session-stateful, the per-call rule breaks — escalate before coding `mcp-client.ts`.*

- **OD-2 — `client_tag` scheme.** UXIE-DISCORD-GUIDELINES.md **line 268** says `crypto.randomUUID()` for slash commands. **Recommendation: edit §268 to the deterministic `uxie-<id>`/`uxie-msg-<id>` scheme** (matches Design + Plan; required for retry-dedupe and cross-plane grep). Verified by grep that this drift lives in Guidelines, not the design spec (the tracking dimension misattributed the file). The existing spec-consistency-report never caught this — add it as a new row.

- **OD-3 — Logger field names + `LOG_LEVEL`.** Guidelines §18 line 410 says `ts`; line 412 says `interaction_id`. The locked Task 4 test + Design §5.2 use `t` and `interactionId`. **Recommendation: edit §18 to `t` and `interactionId` (both, in one pass).** Separately, `LOG_LEVEL` is the one genuine env-schema conflict (§17.1 is locked at 9 fields): **ship an in-code `info`-default gate in v1; defer the env field + §17.1 bump to v1.5.**

- **OD-4 — Command signature shape.** Architecture proposes `ctx.scrypt.{mcp,rest}`; the Plan already injects clients via factory closures (`buildCaptureCommand(rest)`) with `execute(i, ctx)` where `ctx = { clientTag, log }`. **Recommendation: keep the Plan's factory-closure DI (it already achieves mock-without-`mock.module`); optionally fold the client reference into `ctx` for uniformity, but do NOT introduce a competing `execute(i, deps:{rest,mcp})` signature.** (The testing dimension's `execute(i, deps)` rec was rejected for being built on a false premise.)

- **OD-5 — `/ping` MCP probe.** Tracking proposed `mcp.ping()`; Design §6.7 explicitly forbids probing MCP directly. **Recommendation: do NOT add an MCP probe; keep `/ping` REST-only per §6.7.** Only revisit if §6.7 is amended and a true no-op MCP liveness tool exists.

- **OD-6 — `/ping` output: string vs embed.** Plan Task 13's test asserts `editReply` is a STRING matching `/alive.*scrypt: ok/i`; the discord-ux dimension proposed a `pingEmbed`. **Recommendation: keep `/ping` string-based for v1 (matches Design §6.7's literal reply string and the locked test); a richer ping embed is a v1.5 item requiring a test change.**

- **OD-7 — Scrypt web-UI permalink scheme.** `noteUrl()` assumes `/n/<path>`. **Recommendation: confirm the exact route (path vs slug vs note id) against Scrypt before hardcoding;** the permalink is the single highest-value UX touch, so getting the scheme right matters.

- **OD-8 — Timeout drift (Guidelines §12.1 vs Design).** Guidelines §12.1 says "5s default; 10s for `/journal`"; Design §304 says all calls use `AbortSignal.timeout(10_000)` and §295 says health uses 500ms. **Recommendation: adopt Design's values (10s default, 500ms health) and edit Guidelines §12.1 to match;** Design is internally consistent and is what the per-route-timeout test should assert.

- **OD-9 — CHANGELOG ceremony.** Documentation listed `CHANGELOG.md` as mandatory-v1; the verifier demoted it. **Recommendation: make it v1.5/optional** — per-doc revision logs + git tags suffice for a single-owner bot with no release process.

## 6. Completeness gaps for the build phase (what this research did NOT fully cover)

These are the things a build workflow still needs that the six dimensions left thin or as open questions:

1. **Exact Scrypt response contracts (zod schemas).** No dimension produced verbatim zod schemas for `SearchHit[]` / `SemanticHit[]` / `Note` / `DailyContext` / the ingest result. The MCP `tools/call` envelope shape is unconfirmed: is it MCP-standard `{content:[{type:"text",text:"<json>"}]}` (requiring a double-parse) or a structured `result`? This gates `mcp-client.ts` and every parser. **Build must define these from the live Scrypt server before coding the read clients** — and "parse, never cast" with a `content[].text` double-parse fallback.
2. **`/api/ingest` + `/api/daily_context` body/field names.** Design §6 *assumes* `{kind, content, client_tag, tz, meta?}` and a daily-context shape (`today_journal`, `recent_notes`, `open_threads`, `active_memories`, `tag_cloud?`). Unconfirmed. The `tz` field on journal ingest may need a Scrypt-side change (Design §6.5).
3. **Exact embed field layouts per command.** The discord-ux dimension gave builder *sketches* and the canonical names, but the precise field/description/footer layout for `searchResultEmbed`/`semanticResultEmbed`/`briefEmbed` (especially `/brief` multi-section vs single-description) depends on the unconfirmed `daily_context` shape. **Finalize layouts after the contract is known.**
4. **The CI file is described but not written.** The exact `.github/workflows/ci.yml`, `bunfig.toml`, and `package.json` scripts block need to be authored (and reconciled with "v1 has no CI in scope" — treat as an artifact that exists but gates nothing).
5. **The `#inbox` empty/attachment-only path** and **codeblock sanitization** (Guidelines §9) for passive capture are unspecified beyond a mention.
6. **`Bun.Glob` dynamic-import behavior inside `oven/bun:1-alpine`** is unverified — needs a smoke check or a fallback barrel.
7. **Privileged `MessageContent` toggle** sufficiency for an unverified bot ≤100 guilds (surface §7-Q2) — confirm before relying on `#inbox`.
8. **Bun stdout flush-before-exit** behavior in the `process.fatal` handler — verify the fatal line is not dropped (undercuts the logger choice if it is).
9. **Verbatim file-by-file task ordering** — this report is dimension-organized; the build needs the existing 26-task Plan as the execution order, with the new items slotted in (process.on catch site, anti-pattern lint correction, `outcome:"empty"`, rate-limit logging, schema_version field, the five folded-in tests).

## 7. Appendix — Rejected ideas (logged so we remember why)

- **`execute(i, deps:{rest,mcp})` DI signature (testing R1).** REJECTED. Built on the false premise that Plan commands take only `interaction`; the Plan already does DI via factory closures with `execute(i, ctx:{clientTag,log})`. Would replace a working locked pattern. (Use `ctx` extension instead — OD-4.)
- **`mcp.ping()` / MCP liveness probe in `/ping` (tracking R3 half).** REJECTED for v1. Reverses Design §6.7's explicit "no direct MCP probe" decision; streamable-http endpoints don't answer unauthenticated probes and a bounded `searchNotes("",1)` is a real FTS5 read, not a no-op. (OD-5.)
- **`pingEmbed` rich embed (discord-ux).** DEFERRED to v1.5. Contradicts Plan Task 13's string-assertion test. (OD-6.)
- **In-process Prometheus `/metrics` endpoint (tracking).** REJECTED. Mutable cross-interaction counters = stateless breach; inbound port binding violates Tailscale-only posture; aggregation is Para-RAID's chartered job.
- **OpenTelemetry in uxie (tracking).** REJECTED. Native `fetch`/`Bun.serve` not auto-instrumented (all manual spans); pulls 5+ packages + a SIGTERM hook; blows the dep budget for marginal gain over `client_tag`-over-journald.
- **Sentry/GlitchTip (tracking).** REJECTED for v1. Adds a dep + network sink; the three catch sites already log full stacks to journald a single owner tails.
- **pino (logging).** REJECTED. Worker-thread transports broken/fragile on Bun; perf irrelevant; blows the dep budget. Hand-rolled ~40-line logger instead.
- **Embed markdown snapshot tests (testing).** REJECTED for v1 (Design §8.3 forbids). A narrowed `.toJSON()` field snapshot is a v1.5 *candidate* only, contingent on a §8.3 doc bump and evidence the existing field assertions miss regressions.
- **`core/registry.ts` + `Integration` interface (architecture, from ideation brief).** REJECTED. Over-engineering polymorphism for one module; Design §3.2 rejects it; extract `core/` only at module #2.
- **Root `CHANGELOG.md` as mandatory-v1 (docs D8).** DEMOTED to v1.5/optional — ceremony for a single-owner bot with no release process; overlaps per-doc revision logs. (OD-9.)
- **`buildScryptModule().commands` API for `gen-command-docs.ts` (docs D4).** REVISED — consume the builder array `deploy-commands.ts` already assembles; the asserted module API may not exist.
